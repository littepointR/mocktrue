# Examples

PortWeave examples are generated by application code in `frontend/src/workspace/demoWorkspaces.ts`. They are not static fixture files: each load creates a fresh workspace snapshot with generated IDs and `portweave-demo-*` resource names.

Load path in the current UI:

1. Start the app.
2. Open Settings.
3. Open the Global / 全局设置 panel.
4. Select a demo in the “示例配置” selector.
5. Click “加载示例”.
6. Open the loaded graph tab and start the graph runtime manually.

Demos do not auto-start. They create workspace/graph snapshot state; live OS resources are created only after graph runtime start.

## Common expectations

- Demo graph documents should validate before runtime start.
- Runtime resources are graph-owned and created at start time.
- The generated snapshots do not preload serial handles, virtual ports, bridge records, monitor sessions, protocol sessions, or runtime buffers.
- Generated virtual port names are intentionally unique per load.
- No current demo should use removed early-planning graph providers; use current nodes such as `serial.script.generator`, direct output fan-out, `serial.filter`, and `serial.monitor`.

## Demo sections

### `serial-open-demo`

Purpose: a minimal no-hardware graph showing generated traffic entering a virtual serial endpoint.

```text
serial.script.generator.out -> serial.virtual.tx
```

Inspect after start:

- Virtual node endpoint buffer.
- TX/RX counters on the generator and virtual node.
- Graph status/errors.

Expected observation: the script generator writes a short text payload into the virtual endpoint. This is a safe first demo because it does not touch a physical device.

### `virtual-port-demo`

Purpose: create several independent virtual endpoint branches.

```text
serial.script.generator.out -> serial.virtual.tx
serial.script.generator.out -> serial.virtual.tx
serial.script.generator.out -> serial.virtual.tx
```

Inspect after start:

- Each virtual node has a distinct generated `portweave-demo-*` name.
- Branches are independent; there is no shared endpoint identity.
- Runtime status/counters per node.

### `bridge-demo`

Purpose: demonstrate two virtual byte streams connected through `serial.bridge`.

```text
serial.script.generator.out -> serial.virtual.tx
serial.virtual.rx           -> serial.bridge.a-in
serial.bridge.b-out         -> serial.monitor.in

serial.script.generator.out -> serial.virtual.tx
serial.virtual.rx           -> serial.bridge.b-in
serial.bridge.a-out         -> serial.monitor.in
```

Inspect after start:

- Bridge node counters.
- Monitor frames on both sides.
- Any validation errors for bridge handle IDs.

### `monitor-demo`

Purpose: show monitor frames from a byte stream.

```text
serial.script.generator.out -> serial.virtual.tx
serial.virtual.rx           -> serial.monitor.in
```

Inspect after start:

- `serial.monitor` frame list.
- Display mode (`hex` by default for monitor nodes unless demo overrides it).
- Node buffer and frame count.

### `script-transform-demo`

Purpose: show safe script transformation in the middle of a byte path.

```text
serial.script.generator.out -> serial.virtual.tx
serial.virtual.rx           -> serial.script.transform.in
serial.script.transform.out -> serial.monitor.in
```

Inspect after start:

- Transform script config.
- Output bytes from the transform.
- Script timeout/error status if the script fails.

### `script-analyzer-demo`

Purpose: show side-by-side raw monitoring and script field extraction.

```text
serial.script.generator.out -> serial.virtual.tx
serial.virtual.rx           -> serial.monitor.in
serial.virtual.rx           -> serial.script.analyzer.in
```

Inspect after start:

- Monitor frames for raw bytes.
- Analyzer fields emitted through `field()`.
- Analyzer errors if the script cannot parse the payload.

### `modbus-demo`

Purpose: show a Modbus master/slave request-response path without hardware.

```text
serial.modbus.master.tx -> serial.virtual.tx
serial.virtual.rx       -> serial.modbus.slave.rx
serial.modbus.slave.tx  -> serial.modbus.master.rx
serial.modbus.slave.tx  -> serial.monitor.in
```

Inspect after start:

- Modbus master and slave frame views.
- Unit IDs and function code config.
- Monitor bytes for slave responses.

The response edge into `serial.modbus.master.rx` uses the protocol terminal-input cycle exception documented in [Runtime Flow](serial-graph-runtime-flow.md).

### `fecbus-demo`

Purpose: show FECbus master/slave traffic without hardware.

```text
serial.fecbus.master.tx -> serial.virtual.tx
serial.virtual.rx       -> serial.fecbus.slave.rx
serial.fecbus.slave.tx  -> serial.monitor.in
```

Inspect after start:

- Master request frame.
- Slave address/default status config.
- Monitor output from slave responses.

### `serial-graph-demo`

Purpose: demonstrate graph composition and direct output fan-out.

```text
serial.script.generator.out -> serial.virtual.tx
serial.virtual.rx           -> serial.modbus.master.rx
serial.virtual.rx           -> serial.monitor.in
```

Inspect after start:

- One output feeding two downstream inputs.
- Validation result for fan-out.
- Modbus/monitor observations from the same virtual endpoint output.

### `serial-observability-demo`

Purpose: show filters and monitor branches as topology-native observability.

```text
serial.script.generator.out -> serial.virtual.tx
serial.virtual.rx           -> serial.filter.in -> serial.monitor.in
serial.virtual.rx           -> serial.filter.in -> serial.monitor.in
serial.virtual.rx           -> serial.filter.in -> serial.monitor.in
```

Inspect after start:

- Plain filter for `STATUS OK`.
- Regex filter for `TEMP=\d+`.
- Expression filter such as `len >= 4 and text contains "OK"`.
- Monitor frames behind each filter.

### `remote-serial-demo`

Purpose: show `serial.remote` raw TCP client/server behavior entirely inside one graph.

```text
serial.script.generator.out -> serial.remote(client).tx
serial.remote(server).rx    -> serial.monitor.in
serial.remote(client).rx    -> serial.monitor.in
```

Default config uses `protocol = raw-tcp`, `host = 127.0.0.1`, `port = 3001`, one server node and one client node.

Inspect after start:

- Server/client remote node status.
- Endpoint buffers.
- Monitor branches.
- Any `reconnecting` or connection error state.

Security reminder: raw TCP has no built-in authentication, encryption, or serial parameter negotiation. See [Remote Serial Graph Node](remote-serial-node.md).

### `full-workspace-demo`

Purpose: provide one combined workspace with all major no-hardware graph branches.

It includes branches for:

- Virtual endpoint + filter + monitor.
- Bridge with two virtual endpoints.
- Script transform and analyzer.
- Modbus master/slave.
- FECbus master/slave.

Inspect after start:

- Workspace snapshot structure.
- Graph tabs and node tabs.
- Multiple independent branches.
- No preloaded live serial handles or protocol sessions.

See [Workspace and Graph Schema](workspace-graph-schema.md) for file format details.
