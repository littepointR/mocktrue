Phase 2

受控串口资源脚本节点

第二阶段允许脚本访问串口，但访问对象必须由节点配置声明，并由 PortWeave runtime 在启动时授权。脚本不能传路径打开或写任意串口。

节点配置portName、波特率、编码
Runtime 授权冲突检查并打开资源
Script API只写绑定句柄

目标

解决脚本需要模拟真实设备或主动应答外部串口请求的场景，同时保持资源生命周期、冲突检查和拓扑可观察性。

Capability-based

脚本拿到的是 PortWeave 授权能力，不是系统路径。

统一资源所有权

打开、关闭、冲突检查和退出清理都由 graph runtime 负责。

拓扑不失真

数据仍能计数、记录、显示和通过 demo/测试验证。

实现

新增 `serial.script.endpoint`。节点启动时绑定一个串口资源，脚本通过 `serial.*` API 写入该绑定资源。

节点配置

  • `portName`：绑定端口名。
  • `baudRate/dataBits/stopBits/parity/flowMode`。
  • `readBufKB` 和 `encoding`。
  • `script`、`timeoutMs`、`maxOutputBytes`。

新增 API

serial.write(bytes)
serial.writeHex("01 02")
serial.writeText("OK\r\n", "utf-8")
serial.info()

运行流程

endpoint 节点的资源绑定发生在图启动阶段，脚本执行阶段只能使用已绑定句柄。

Step 1

校验配置

确认 portName 和串口参数有效。

Step 2

冲突检查

复用现有串口、监控、桥接、协议占用检查。

Step 3

打开资源

runtime 打开端口并记录 owner。

Step 4

执行脚本

脚本只看到 serial capability。

Step 5

读写路由

RX/TX 进入计数和 buffer。

Step 6

统一清理

停止图时关闭资源。

禁止行为

这些限制是 endpoint 能安全落地的前提。

不允许 `serial.open("/tmp/ttyX")`、`serial.write("/dev/cu.usbserial", bytes)`、文件读写、网络请求、进程执行、环境变量访问和动态模块加载。

验收

endpoint 的验收重点是资源安全和生命周期。

必须通过

  • 启动时占用检查有效。
  • 脚本只能写当前绑定资源。
  • 停止图后端口关闭。
  • 程序退出后没有虚拟串口残留。

主要风险

  • 隐式回灌循环。
  • 脚本绕开拓扑写数据。
  • 端口重复占用。
  • 错误路径下资源未清理。
