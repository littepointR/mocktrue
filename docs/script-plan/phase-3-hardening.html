Phase 3

增强、稳定化和长期运行验证

第三阶段的目标是把脚本能力从“可用”打磨到“长期可靠”。重点是模板、诊断、压力测试、日志和用户体验，而不是扩大系统权限。

A

模板

常用协议脚本

B

诊断

错误、耗时、输出

C

压测

长时间满负载

D

体验

补全和文档

目标

提升脚本功能的可维护性和可诊断性，让用户能快速编写、复用和定位问题。

模板库

提供 Modbus、FECbus、自定义帧、CRC、循环数据等脚本模板。

可观测性

展示执行耗时、错误次数、最后输出、字段表和日志。

稳定性

覆盖长时间运行、最小间隔、停止恢复、demo 切换和资源回收。

实现

增强围绕现有节点展开，不改变安全边界，不新增任意系统访问能力。

功能增强

  • 脚本模板选择和插入。
  • API 文档面板。
  • 执行日志和错误历史。
  • 字段表导出到监控视图。

稳定化

  • 压力测试脚本集。
  • demo 自动启动验证。
  • 内存和 goroutine 泄漏检测。
  • 错误恢复和节点重启策略。

流程

每个增强先以真实用户场景定义模板，再补测试和文档。

Step 1

收集场景

从串口、Modbus、FECbus 和自定义协议中选高频场景。

Step 2

写模板

模板必须只使用公开脚本 API。

Step 3

补提示

Monaco completion 和 hover 同步更新。

Step 4

跑压测

最小间隔、长时间、切换 demo。

Step 5

固化验收

把发现的问题转成自动化测试。

验收

第三阶段完成后，脚本功能应适合真实项目长期使用。

质量标准

  • 所有 demo 启动后都有循环数据。
  • 100 秒最小间隔压测无卡死。
  • 停止后资源和计数稳定。
  • 脚本错误可定位到具体节点和行。

不做的事

  • 不扩大脚本系统权限。
  • 不让脚本绕过拓扑资源管理。
  • 不引入无法自动验证的 demo 特殊逻辑。
  • 不把协议 manager 直接嵌入脚本节点。
