Workflow 1

需求分析阶段

目标是把脚本功能的边界、节点类型、API、UI 行为和验收标准锁定到实现者不需要再做产品判断。

阶段目标

需求分析 agent 只读，不写代码。输出的是可执行规格，而不是实现方案草稿。

确认边界

明确脚本只能通过 PortWeave API 工作，不能直接访问系统资源。

确认节点

锁定 transform、generator、analyzer 和后续 endpoint 的职责。

确认验收

把 demo、保存恢复、MCP、压力测试都纳入完成定义。

步骤

按从产品到技术边界的顺序整理，避免实现阶段反复改方向。

Step 1

读取现状

确认现有 graph、script engine、MCP、demo、测试约束。

Step 2

定义角色

明确每个脚本节点的输入、输出和禁止能力。

Step 3

定义 API

列出公开 API、默认配置和限制。

Step 4

定义验收

用可自动测试的场景描述完成条件。

Step 5

交付规格

给后端、前端、MCP/demo agent 使用。

输出物

需求阶段完成后，后续 agent 不需要猜测范围。

输出内容用途
节点规格类型、端口、默认配置、显示能力前后端 provider 同步
API 规格函数签名、可用节点、禁止调用runtime 和 Monaco 补全
验收矩阵后端、前端、MCP、demo、压力测试验证阶段执行
风险清单越权、资源泄漏、循环、运行态一致性审查阶段重点
