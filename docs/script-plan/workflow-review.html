Workflow 5

审查阶段

目标是从风险角度确认脚本功能没有破坏拓扑语义、资源生命周期、安全边界和用户配置一致性。

审查目标

审查 agent 只读，不修复。发现问题后由对应实现 agent 处理。

安全

脚本不能越权访问系统能力或任意串口路径。

生命周期

图停止和程序退出时没有任务、端口或虚拟串口残留。

一致性

UI、MCP、runtime、demo 的 provider 和配置一致。

审查清单

按风险排序，优先处理可能导致越权、残留或数据错路由的问题。

类别检查点阻塞条件
脚本沙箱是否开放文件、网络、进程、env、动态模块任何开放都阻塞
资源所有权endpoint 是否统一打开和关闭端口脚本可传路径则阻塞
运行时超时、取消、定时器停止是否完整停止后仍发送则阻塞
拓扑脚本输出是否全部经过端口和连接线绕过拓扑写数据则阻塞
前端Monaco 诊断是否覆盖禁用 API无提示但 runtime 禁止则需修复
保存恢复源码、配置、布局、历史是否 round trip配置丢失则阻塞

流程

审查要在 targeted tests 通过后进行，避免浪费在明显未完成的变更上。

Step 1

读 diff

确认变更范围和风险点。

Step 2

查安全

检查脚本 API 和 runtime 暴露。

Step 3

查生命周期

停止、错误、退出路径。

Step 4

查一致性

UI、runtime、MCP、demo 同步。

Step 5

列发现

按严重度输出文件和行号。
