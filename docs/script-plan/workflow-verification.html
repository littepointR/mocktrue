验证阶段

目标是用自动化证明脚本功能真实有效。验证不靠截图观察，而是通过 runtime 状态、计数、buffer、frame 和测试断言完成。

验证目标

每个目标都要有命令和可判定结果，避免“看起来能用”。

功能正确

脚本节点按 API 处理数据，错误路径可控。

集成正确

保存恢复、MCP、demo 和 build 都通过。

运行稳定

最小间隔和长时间运行不会卡死或泄漏。

验证步骤

从窄到宽，先定位问题，再跑完整验证。

Step 1

Targeted Go

脚本 runtime 和 graph tests。

Step 2

Targeted Vitest

ScriptEditor、SerialGraphPanel、workspace。

Step 3

类型检查

vue-tsc noEmit。

Step 4

构建

Step 5

Demo 压测

自动启动并读取状态断言计数增长。

验收命令

具体命令在实现后可按实际包路径微调，但验证范围不能缩小。

go test ./internal/modules/serial/... ./internal/modules/mcpserver/...
cd frontend
pnpm exec vitest run src/serial/graph/serialGraph.test.ts
pnpm exec vitest run src/serial/stores/graphStore.test.ts
pnpm exec vitest run src/serial/views/SerialGraphPanel.test.ts
pnpm exec vitest run src/workspace/demoWorkspaces.test.ts
pnpm exec vitest run src/workspace/workspaceSession.test.ts
pnpm exec vue-tsc --noEmit
pnpm run build:dev

自动化场景

对 demo 的验证要读取程序状态，不采用人工截图判断。

Demo 断言

  • 加载每个 demo 得到不同拓扑。
  • 启动后 10 秒内 RX/TX 增长。
  • 查询 receiver buffer 能读到数据。
  • analyzer frame 总数增长。

压力断言

  • 100 秒最小间隔运行不中断。
  • 停止按钮返回成功。
  • 停止后计数保持不变。
  • 重复启动停止没有资源残留。
